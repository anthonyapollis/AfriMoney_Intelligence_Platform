1.1 The Business Problem

South Africa hosts an estimated 3.5 million foreign nationals who send money home regularly to Zimbabwe, Mozambique, Zambia, Malawi, and beyond. The SA–Africa remittance market moves billions of rand monthly, yet most operators run on siloed data systems with no unified customer view.

AfriMoney models two leading operators — Mukuru (Africa's largest MTO, 17M customers) and Mama Money (wallet-first fintech, ISO 9001 certified) — on a single Snowflake data platform. One BUSINESS_KEY column (MKR / MMY) separates the brands; every mart, every ML model, every KPI covers both.

2.2 ETL — COPY INTO from Internal Stage

-- Step 1: create file format CREATE OR REPLACE FILE FORMAT AFRIMONEY_CSV_FORMAT TYPE = CSV FIELD_OPTIONALLY_ENCLOSED_BY = '"' NULL_IF = ('NULL','\N','') EMPTY_FIELD_AS_NULL = TRUE SKIP_HEADER = 1 DATE_FORMAT = 'YYYY-MM-DD'; -- Step 2: upload to stage PUT file://data/bronze/fact_remittance_transfer.csv @STAGING.AFRIMONEY_STAGE AUTO_COMPRESS = TRUE; -- Step 3: load Bronze table USE WAREHOUSE AFRIMONEY_TRANSFORM_WH; COPY INTO BRONZE.FACT_REMITTANCE_TRANSFER FROM @STAGING.AFRIMONEY_STAGE/fact_remittance_transfer.csv.gz FILE_FORMAT = (FORMAT_NAME = AFRIMONEY_CSV_FORMAT) ON_ERROR = 'CONTINUE'; -- Step 4: verify load SELECT COUNT(*), MIN(_LOADED_AT), MAX(_LOADED_AT) FROM BRONZE.FACT_REMITTANCE_TRANSFER; -- → 5,000,000 rows loaded in ~65s
Why internal stage? No external cloud storage account needed for a portfolio setup. Data is uploaded once via PUT, Snowflake compresses it automatically, and COPY INTO is idempotent — re-running it skips already-loaded files based on load metadata.

3.1 Why dbt?

dbt compiles SQL .sql files into CREATE TABLE / VIEW AS SELECT statements and runs them in DAG order. Every model is Git-versioned, auto-tested, and documented. When a business analyst asks "where does the revenue number come from?" the answer is traceable to a Bronze CSV row.

Business-ready tables for Power BI, Streamlit, and ML feature extraction

4.1 mart_remittance

Grain: business_key + corridor_code + month + channel + payment_method. Pre-computes every KPI an executive needs — funnel rates, revenue splits, FX margins, fraud basis points.

SELECT business_key, corridor_code, created_month, success_rate_pct, total_net_revenue_zar, fraud_rate_bps, avg_fx_spread_pct FROM GOLD.MART_REMITTANCE WHERE created_month >= '2026-01' ORDER BY total_net_revenue_zar DESC;

4.2 mart_customer_360

One row per current customer. Derives two composite scores entirely in SQL:

  • LTV Score — lifetime_revenue × 0.60 + card_spend × 0.002 + transfer_count × 5 + wallet_balance × 0.01
  • Engagement Score (0–100) — recency weight × 40 + frequency × 30 + KYC completion × 20 + wallet active × 10

Training environment: Models trained in Python (sklearn) on 160,000 + 40,000 + 16,000 rows of synthetic fintech data mirroring the Snowflake schema. Production equivalents run via Snowpark ML — identical algorithms, data never leaves Snowflake.

Model 1 — Fraud Detection  |  GradientBoostingClassifier

What it does: A Gradient Boosting ensemble trained to flag transfers likely to be fraudulent at the moment of initiation. Outputs a probability 0–1; transfers above 0.40 are held for compliance review before processing. GBM is chosen over logistic regression because fraud patterns are highly non-linear — high send amounts are only suspicious when combined with multiple payment attempts and unusual hours.

Reading the results

AUC 0.9582 — the model correctly ranks a random fraudulent transfer above a random legitimate one 95.8% of the time. An AUC of 0.50 = random guessing; 1.00 = perfect.

Top features: Payment attempts and send amount log dominate — a customer who retries payment 3+ times on a large transfer is the clearest fraud signal.

Threshold at 0.40 — set below 0.50 to catch more fraud (higher recall), accepting some false positives that require manual review.

What it does: A Random Forest trained on customer behavioural features to predict who will stop sending transfers in the next 90 days. Scores are computed weekly in batch. Customers in the top two churn risk bands get targeted CRM campaigns — push notifications, fee discounts, or WhatsApp reminders.

Reading the results

AUC 0.9905 — strong predictive power on customer churn. The model's confidence is consistent across all 5 CV folds (low variance = no overfitting).

Days since last transfer dominates feature importance — recency is the single strongest churn signal. Monthly transfer rate and engagement score round out the top 3.

RF chosen here (vs GBM) because it handles high-dimensional customer data with mixed feature scales robustly, and its built-in bagging reduces variance on the 10.9% churn rate dataset.

Model 3 — Credit Risk PD  |  GradientBoostingClassifier (Mukuru only)

What it does: Outputs a Probability of Default (PD) used for two purposes: (1) origination decisions — applications above the PD threshold are declined; (2) IFRS 9 Expected Credit Loss (ECL) — ECL = PD × LGD × EAD, with a 45% Loss Given Default assumption. SHAP explanations are generated for every declined application for NCA compliance.

Reading the results

AUC 0.9475 — solid for a credit risk model. Industry benchmark for consumer lending PD models is 0.70–0.80 AUC; we are in range.

DTI ratio (debt-to-income) and previous DPD (days past due) dominate. These are the classic IFRS 9 Stage 2 indicators — a borrower with prior delinquency is 3× more likely to default.

IFRS 9 ECL: avg_pd × avg_loan_size × 0.45 LGD, pre-calculated in mart_loans_mukuru and refreshed daily.

Snowpark ML & Model Registry
Running the same models inside Snowflake — zero data egress, full governance

6.1 Why Snowpark?

Traditional ML pulls data out of the warehouse to a Python server, trains, then pushes predictions back. That means egress costs, PII exposure, and pipeline fragility. Snowpark ML runs Python code inside Snowflake compute — the feature tables, training jobs, model registry, and prediction tables are all Snowflake objects governed by the same RBAC roles as your SQL queries.
